## 记录格式规范

每条错误记录必须包含以下部分：

### 1. 文件命名

```
YYYY-MM-DD.md  # 例如：2026-04-25.md
```

### 2. 文档结构

```markdown
# 错误日志 - YYYY年MM月DD日

---

## [HH:MM] ❌ 错误类型：问题简述

**严重程度**：高/中/低 | **状态**：✅ 已修复/🔄 处理中

### 问题摘要
一句话说明发生了什么问题。

### 前因
为什么会出现这个问题？

### 经过
1. 第一步...
2. 第二步...

### 根因

| 维度 | 分析 |
|------|------|
| 直接原因 | ... |

### 改进措施

1. ...

### 相关文件

```
file1.tsx  # 说明
file2.css  # 说明
```
```

### 3. 标题格式要点

- **时间戳**：`[HH:MM]` 放在最前面，一眼看出是几点的问题
- **状态图标**：`❌` 表示错误，`✅` 表示已修复
- **问题类型**：记忆偏差、遗漏检查、想当然等
- **问题简述**：一句话说明

---

## 根因分类

| 分类 | 说明 |
|------|------|
| **记忆偏差** | 混淆了变量名、函数名、路径等 |
| **遗漏检查** | 改了一处忘记检查相关处 |
| **上下文丢失** | 未仔细阅读现有代码就做判断 |
| **想当然** | 凭直觉而非验证就行动 |

---

## 改进原则

1. **先读后改** - 修改前必须先完整阅读相关代码
2. **全面搜索** - 改了一处，搜索同类文件是否有相同模式
3. **验证假设** - 对不确定的内容要实际验证
4. **及时复盘** - 错误后及时记录分析
