# 本地开发数据库安全规范

## 1. 背景

本地开发数据库文件（如 `.local_dev.sqlite3`）可能包含敏感信息，如用户密码、API 密钥等。为了保护这些信息不被泄露，需要建立严格的安全规范。

## 2. 禁止事项

- **绝对禁止**将本地开发数据库文件提交到 git 仓库
- **绝对禁止**在代码中硬编码数据库连接信息
- **绝对禁止**在公共场合分享包含敏感信息的数据库文件

## 3. 最佳实践

### 3.1 添加到 .gitignore

确保本地开发数据库文件已添加到 `.gitignore` 文件中：

```gitignore
# 本地开发数据库
backend/.local_dev.sqlite3
```

### 3.2 定期清理

- 开发完成后，及时删除本地开发数据库文件
- 切换分支前，确保清理本地数据库文件
- 定期检查项目根目录，确保没有遗漏的数据库文件

### 3.3 安全检查

- 使用 `git status` 命令定期检查未提交的文件
- 使用 `git log --all --name-only --grep=.sqlite3` 检查是否有数据库文件被提交过
- 确保所有数据库连接信息都通过环境变量配置

## 4. 紧急处理

如果发现本地数据库文件已被提交到 git 仓库：

1. **立即停止**任何代码推送操作
2. **使用 git filter-branch** 或 BFG Repo-Cleaner 工具从历史记录中删除该文件
3. **修改**所有可能被泄露的敏感信息（如密码、API 密钥）
4. **通知**团队成员，确保他们也更新相关信息

## 5. 验证方法

- 执行 `git status` 确认没有未提交的数据库文件
- 执行 `git log --all --name-only --grep=.sqlite3` 确认没有数据库文件的提交记录
- 检查 `.gitignore` 文件，确保包含本地开发数据库文件

## 6. 总结

保护本地开发数据库文件是项目安全的重要组成部分。通过严格遵守本规范，可以有效防止敏感信息泄露，保障项目的安全性。
