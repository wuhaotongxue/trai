## 核心规则

### 1. 中文标点禁令 (CRITICAL)
- **绝对禁止**在代码和注释中出现中文全角标点 (`，`、`。`、`！`、`：`)
- 必须使用英文半角标点 (`,`, `.`, `!`, `:`)

### 2. Python_3_13 环境
- 标准环境：`trai31313` (Conda)
- 强制使用 `|` 替代 `Union`/`Optional`
- 缩进 4 个空格

### 2.1 强制动作自查表 (MANDATORY)

- **[ ] 文件头**: 是否包含作者(wuhao)、日期、描述？
- **[ ] 类封装**: 是否所有函数都封装在类中？(禁止孤立函数)
- **[ ] Docstring**: 每个 class 和 def 是否都有详细的中文文档？
- **[ ] 类型提示**: 是否使用 `|` 和 `list[]` 等 Python 3.13 语法？
- **[ ] 日志**: 是否严禁 `print()`，统一使用 `loguru`？
- **[ ] 标点**: 是否通过正则检查，排除了所有中文全角标点？

### 3. 强制类封装与 DDD 分层
- **每个 .py 文件必须有类**
- **所有函数必须封装在类中**
- 禁止文件顶层孤立函数

### 4. API 设计规范
- 业务接口强制仅使用 **POST**
- 统一响应格式: `{"code": 200, "msg": "OK", "data": {}, "req_id": "...", "ts": "..."}`

### 5. 数据库规范
- 表名: 复数 snake_case
- 主键: UUID 或 BigInt Identity (禁止 Serial)
- 必备字段: `created_at`, `updated_at`, `is_deleted`
- **强制注释 (MANDATORY)**: 所有 Table 和 Column 必须有 `comment` 描述，严禁无注释字段。

### 5.1 通用任务流规范 (TaskFlow)
- **固化流程**: 所有涉及文件处理的任务必须遵循 `上传 S3` -> `记录数据库` -> `多端通知 (飞书/企微)` 的标准流程。
- **通用服务**: 强制调用 `TaskFlowService` 实现上述逻辑，禁止在业务 Router 中重复编写上传/通知代码。
- **通知内容**: 必须包含处理结果预览链接 (Presigned URL) 和 AI 专家点评。

### 6. 文件头模板 (MANDATORY)
```python
#!/usr/bin/env python
# _*_coding:_utf_8_*_
# 文件名:_{实际文件名}
# 作者:_wuhao
# 日期:_{YYYY_MM_DD_HH:MM:SS}
# 描述:_{该文件的用途/功能简述，一句话概括}
```

#### env 文件夹下配置文件模板 (如 notify_robot.env)

> **注意：** 环境变量已不再使用单一的 `.env` 文件，而是按模块拆分到 `backend/env/` 目录下（例如 `notify_robot.env`, `llm.env` 等）。修改或新增配置必须在此目录下进行。

```bash
# ============================================================
# TRAI Backend Environment Configuration
# 文件名: {对应的.env文件名，如 notify_robot.env}
# 作者: wuhao
# 日期: 2026_04_09_10:47:12  ← 当前时间（每次修改配置时刷新）
# 描述: {配置文件用途简述}
# 说明: {补充说明}
# ============================================================
```

### 7. Docstring 全覆盖
- 每个 `class`、每个 `def`/`async def` 必须有中文 docstring
- 内容: 用途 + 参数 + 返回值 + 异常

### 8. 安全红线
- **禁止硬编码绝对路径**：严禁在代码中写死如 `/home/qyjgylc_whf/code/trai/` 等本地绝对路径。所有路径必须基于项目根目录动态计算（如使用 `Path(__file__).resolve().parents[n]`）或从环境变量/配置文件中读取。
- **强制自查导入符号**：在使用 `Path`, `datetime`, `uuid` 等标准库符号前，必须确保文件顶部已正确 `import`。严禁因漏写导入导致 `NameError`。
- **节日仪式感规范**：所有向飞书/企业微信推送的媒体通知，必须优先调用 `FestivalManager` 检查当日是否有节日。若有节日，必须通过 `DeepSeek` 动态生成不重复的文案进行推送，增加系统的人文关怀。
- **推送多样化红线**：禁止发送内容完全重复的推送文案。对于节日推送，必须结合“节日背景+作品内容+随机创意”通过 LLM 生成动态内容，确保用户每次收到的祝福都是独一无二的。
- **提示词工程优化**：在音频/图像生成接口中，应具备基础的 Prompt 增强逻辑（如检测到女性姓名时自动补充 `female singer` 标签），确保生成结果符合预期。
- **多模态完整链路规范**：在音乐/视频生成等复杂任务中，应优先构建完整体验，如音乐生成必须包含“音频+动态歌词+封面图”的全套输出，提升产品的专业度与用户体验。
- **环境自动修复机制**：对于本地模型依赖（如 Sambert 依赖的 `kantts`），应在代码中实现检测与自动 `pip install` 的修复逻辑，最大限度保证数字人等 AI 服务的可用性。
- **禁止硬编码密码**：所有数据库密码、API 密钥必须通过环境变量读取，禁止写死在代码中。
- **禁止使用 eval**：处理用户输入的数学表达式或其他逻辑时，必须使用安全的解析器（如 `ast.literal_eval` 或自定义 AST 遍历），禁止使用内置 `eval()` 防范注入。
- **强制配置强密钥**：AES 加密和 JWT 签名必须在生产环境中配置强随机密钥（>=32字节），如果未配置必须抛出异常阻断启动，禁止静默使用弱默认密钥。
- **数据库连接安全**：手动获取的数据库 Session 必须使用 `with` 上下文管理器或 FastAPI `Depends(get_db_session)` (基于 yield)，确保用完即释放，严防出现 `idle in transaction` 锁死数据库。
- 必须使用 `passlib` + `bcrypt`/`Argon2` 哈希密码
- 严禁在日志中记录明文密码或 Token
- **严禁 S3 资源 Public_Read**，必须通过 Presigned_URL 访问 (详见 `storage/s3_access.md`)
- **企业微信禁止用户名密码登录**，仅支持扫码登录 (SSO)，否则视为普通用户
- **企业微信扫码登录自动授予 `wecom_vip` 权限**（等同于 VIP 会员有效期）
- **EXE 更新不影响 S3 权限**，权限控制仅在 Backend 侧
- **多租户强制隔离**：每次 S3 访问必须校验 `tenant_id`，禁止跨租户访问
- **上传必须双重校验**：Content_Type + 扩展名同时通过才允许上传
- **可执行文件禁止上传**：`.exe/.sh/.py/.js/.jar/.dll/.so` 及对应 MIME 类型一律拒绝
- **下载必须限流**：每次签发 Presigned_URL 前先检查滑动窗口限流
- **S3 操作全量审计**：GetObject/PutObject/DeleteObject 全部记录结构化日志
- **AI 内容必须审核**：文生图/图生视频生成后必须经过内容安全审核才能展示
- **AI 服务直传安全**：Presigned_PUT_URL 有效期 ≤ 300 秒，校验 x_amz_expected_bucket_owner
- **AI 月配额控制**：用户提交 AI 任务前必须校验月度配额，超额拒绝
