# Word 试卷转钉钉答题表单实现方案

## 1. 需求理解

你的目标不是简单上传一个 Word 供人查看, 而是把 `docx` 里的课程/考试内容自动转换成钉钉里的可填写表单, 再生成分享链接给用户答题。

结合你提供的文件 `7-CAP产品知识测试B卷.docx`, 需求可以拆成 4 步:

1. 用户上传 Word 试卷。
2. 后端把 Word 解析成结构化题库数据。
3. 调用钉钉能力, 自动生成类似截图 2 的答题表单页面。
4. 发布分享链接, 用户打开链接后答题并提交。

## 2. 先说结论

这件事可以做, 但推荐你走 **钉钉宜搭 + 结构化题库 + 页面/表单生成** 方案, 不要走普通 OA 审批接口。

更准确地说:

- 如果你要的是“收集用户填写结果”, 钉钉宜搭的数据表单能力是对口的。
- 如果你要的是“根据 Word 自动创建一整张新表单”, 适合走宜搭表单 Schema 生成链路。
- 如果你要的是“用户通过分享链接在线做题”, 适合再加一层宜搭页面发布或公开访问配置。

不建议直接把这个需求建成 OA 审批流, 因为审批流适合“发起审批”, 不适合“试卷答题”。

## 3. 这份 Word 的题目结构

我已经读取了你给的 `docx`, 结构大致如下:

- 试卷标题: `痰热清注射液销售代表专业考核试卷`
- 适用岗位: `学术 / 销售代表`
- 满分: `100`
- 考试时间: `60 分钟`
- 单选题: `10` 题
- 多选题: `15` 题
- 判断题: `15` 题
- 简答题: `1` 题
- Word 末尾自带参考答案和评分标准

这意味着它非常适合先转成统一 JSON, 再映射到钉钉题型。

## 4. 推荐的两种实现方式

### 方案 A: 每上传一份 Word, 自动创建一张新的宜搭答题表单

这个方案最接近你截图 2 的效果。

流程如下:

1. 上传 `docx`
2. 解析出题目、选项、标准答案、分值
3. 调用宜搭表单创建能力, 动态生成字段
4. 发布页面
5. 生成分享链接
6. 用户答题
7. 后端回收答卷并计算成绩

优点:

- 和你截图中的“真实表单”最像
- 每一套卷子都是独立页面, 方便分享
- 不需要前端自己写复杂答题渲染器

缺点:

- 每上传一套新卷, 都要新建一张表单
- 题量大时, 表单 Schema 会比较长
- 如果后续要经常改卷, 维护多套表单会越来越重

### 方案 B: 固定一个“答题页面”, Word 只生成题库数据

这是更推荐的生产方案。

流程如下:

1. 上传 `docx`
2. 后端解析为 `exam_paper + questions + answers`
3. 写入宜搭数据表, 或写入你自己的数据库
4. 一个固定的“答题页面”根据 `paper_id` 动态渲染题目
5. 生成分享链接, 如 `.../exam?paper_id=xxx`
6. 用户答题提交
7. 系统自动判分并保存成绩

优点:

- 不需要每次新建表单 Schema
- 支持无限套卷
- 更适合随机组卷、限时考试、重考、排行榜、成绩分析

缺点:

- 比纯表单多一层页面开发
- 前期开发比方案 A 稍重

## 6. Word 转结构化 JSON 的核心设计

无论走方案 A 还是方案 B, 第一步都一样: 先把 Word 解析成标准 JSON。

建议的中间结构如下:

```json
{
  "paper_title": "痰热清注射液销售代表专业考核试卷",
  "position": "学术 / 销售代表",
  "total_score": 100,
  "duration_minutes": 60,
  "sections": [
    {
      "section_type": "single_choice",
      "section_title": "单项选择题",
      "score_per_question": 2,
      "questions": [
        {
          "question_no": 1,
          "stem": "中国成人 CAP 年发病率为",
          "options": [
            {"key": "A", "text": "7.13/1000"},
            {"key": "B", "text": "9.32/1000"},
            {"key": "C", "text": "14.98/1000"},
            {"key": "D", "text": "5.76/1000"}
          ],
          "answer": ["A"]
        }
      ]
    }
  ]
}
```

### 解析时要解决的几个问题

#### 6.1 题号和题干分离

例如:

`1. 中国成人 CAP 年发病率为（A）`

要拆成:

- `question_no = 1`
- `stem = 中国成人 CAP 年发病率为`
- `answer = ["A"]`

#### 6.2 选项行合并

例如 Word 里常见这种格式:

`A. 65 岁以上老年人 B. 5 岁以下婴幼儿 C. 合并 COPD、糖尿病患者`

需要用正则把一行中的多个选项拆开。

#### 6.3 多选答案提取

例如:

`CAP 临床高危人群包括（ABCDE）`

要解析为:

```json
["A", "B", "C", "D", "E"]
```

#### 6.4 判断题提取

例如:

`CURB-65 评分≥3 分建议门诊居家治疗即可。（）×`

建议统一转为:

```json
{
  "question_type": "judge",
  "answer": ["false"]
}
```

#### 6.5 简答题单独标记

简答题不能像选择题一样自动评分, 需要:

- 人工评分
- 或先只做采集, 不自动判分

## 7. 钉钉宜搭字段映射

如果你走方案 A, 可以这样映射:

| Word 题型 | 宜搭字段类型 | 说明 |
|---|---|---|
| 姓名 | `TextField` | 必填 |
| 部门 | `TextField` 或 `SelectField` | 可选下拉 |
| 单选题 | `RadioField` | 每题一个单选组件 |
| 多选题 | `CheckboxField` 或 `MultiSelectField` | 更推荐 `CheckboxField` |
| 判断题 | `RadioField` | 选项固定为“正确/错误” |
| 简答题 | `TextareaField` | 长文本作答 |
| 分数 | 不放在答题表单里 | 后端计算后回写 |
| 标准答案 | 不暴露给前端 | 存在后台或题库表 |

对你这份卷子, 最终表单大概会长成:

- 基本信息
- 姓名
- 部门
- 单选题 1 ~ 10
- 多选题 1 ~ 15
- 判断题 1 ~ 15
- 简答题 1
- 提交按钮

## 8. 钉钉侧到底用什么能力

这里要分开说。

### 8.1 用于“收集答卷数据”

钉钉宜搭提供了表单实例新增/更新能力, 适合提交用户答卷数据。

官方文档可参考:

- [新增或更新表单实例](https://open.dingtalk.com/document/development/add-or-update-form-instances)
- [获取表单内的组件信息](https://open.dingtalk.com/document/development/get-form-field-information-based-on-form-uuid)

从官方文档可知, 宜搭侧成熟稳定的公开能力重点在:

- 获取表单字段定义
- 新增或更新表单实例
- 查询表单数据

### 8.2 用于“自动生成表单页面”

如果你的目标是“像截图 2 那样自动创建字段”, 更适合走宜搭表单 Schema 生成链路。

这部分更接近宜搭开发侧能力, 常见做法是:

- 先创建应用
- 再创建/更新表单页面
- 再发布页面
- 再配置公开访问或分享

可参考 openyida 的工具链思路:

- [openyida 总入口说明](https://github.com/openyida/openyida/blob/ca45f5e6a854a0f84d208a3e541a8582e5089c45/yida-skills/SKILL.md)
- [表单创建/更新能力说明](https://github.com/openyida/openyida-openclaw-skill/blob/2f535c78e6c7ec84ab2d3c3fd10e0820a8118cf7/skills/yida-create-form-page/SKILL.md)

一句话总结:

- “提交答卷数据”这部分, 钉钉开放文档比较明确
- “自动搭一整张表单页面”这部分, 更像宜搭开发链路, 不建议你拿 OA 审批接口去硬做

## 9. 推荐的系统架构

建议后端拆成以下模块:

### 9.1 `word_parser`

职责:

- 读取 `docx`
- 提取试卷标题、考试时间、总分、题型分组
- 解析题号、题干、选项、答案、分值
- 输出标准 JSON

建议技术:

- Python: `python-docx`
- 正则: `re`

### 9.2 `exam_normalizer`

职责:

- 统一题型名称
- 清洗换行、全角括号、残缺答案标记
- 校验题号连续性
- 校验选项完整性

### 9.3 `dingtalk_yida_client`

职责:

- 获取 access token
- 调用宜搭应用/表单相关接口
- 保存表单标识
- 发布页面
- 获取分享链接

### 9.4 `exam_scoring_service`

职责:

- 自动判分单选、多选、判断
- 记录简答题待人工评分
- 汇总总分

### 9.5 `share_link_service`

职责:

- 生成分享链接
- 可选加密签名
- 控制有效期
- 限制重复提交

## 10. 具体实施步骤

## 第一步: 做 Word 上传接口

输入:

- `docx` 文件
- 试卷名称
- 是否公开分享
- 是否允许重复提交

输出:

- `paper_id`
- 解析结果预览

## 第二步: 做 Word 解析

把 Word 解析成标准 JSON, 同时做一轮预检查:

- 是否识别到标题
- 是否识别到每个分组
- 是否识别到题号
- 是否提取到答案
- 是否发现异常题目

建议这一步先返回一个“解析预览页”, 让运营确认再生成正式表单。

## 第三步: 生成宜搭字段定义

把题库 JSON 转成字段列表, 例如:

```json
[
  {"type": "TextField", "label": "姓名", "required": true},
  {"type": "TextField", "label": "部门", "required": true},
  {
    "type": "RadioField",
    "label": "1. 中国成人 CAP 年发病率为",
    "required": true,
    "options": ["A. 7.13/1000", "B. 9.32/1000", "C. 14.98/1000", "D. 5.76/1000"]
  },
  {
    "type": "CheckboxField",
    "label": "11. CAP 临床高危人群包括",
    "required": true,
    "options": ["A. 65 岁以上老年人", "B. 5 岁以下婴幼儿", "C. 合并 COPD、糖尿病患者"]
  }
]
```

## 第四步: 创建表单/页面

如果你用宜搭:

1. 创建应用
2. 创建表单页面
3. 写入表单 Schema
4. 发布页面
5. 配置分享访问

如果短期内暂时拿不到完整的“自动创建表单页面”能力, 也可以折中:

1. 先人工创建一个题库模板应用
2. 后端只负责往题库和答卷表里写数据
3. 答题页用自定义页面渲染

这条路更稳。

## 第五步: 生成分享链接

你最终要给用户的不是接口地址, 而是答题链接。

建议链接长这样:

```text
https://your-domain/exam/share?paper_id=PAPER_20260602_xxx&token=xxxx
```

如果你完全依赖钉钉宜搭页面, 则分享的就是宜搭页面地址。

如果你自己做一层前端, 则分享的是你自己的 H5 答题页, 页面里再调钉钉/后端接口拿题。

## 11. 判分建议

### 自动判分

适用于:

- 单选题
- 多选题
- 判断题

规则:

- 单选: 用户答案与标准答案完全一致即得分
- 多选: 必须完全一致才得分
- 判断: `true/false` 完全一致得分

### 人工判分

适用于:

- 简答题

建议:

- 提交后先给出“客观题分数”
- 简答题待老师或管理员评分后再出总分

如果你要自己开发, 推荐技术栈如下:

## 15. 关键风险

### 风险 1: Word 格式不统一

不同老师出的 Word 题目排版可能不同, 导致解析失败。

解决:

- 先制定 Word 模板规范
- 做“解析预览 + 人工确认”

### 风险 2: 自动生成表单页面能力边界

“表单数据写入”是成熟能力, 但“完全自动搭好一张复杂答题页”更依赖宜搭开发链路。

解决:

- MVP 先用模板表单或固定答题页
- 完整自动化再接入宜搭 Schema 生成

### 风险 3: 分享链接权限

如果分享对象不在钉钉组织内, 要确认页面是否支持公开访问。

解决:

- 明确是“组织内答题”还是“外部公开答题”
- 对应配置组织内访问或公开分享页面

### 风险 4: 成绩安全

标准答案不能暴露在前端。

解决:

- 答案只放后端或管理表
- 前端只拿题目和选项

## 16. 最实用的落地顺序

建议你按下面顺序做:

1. 先做 `Word -> JSON`
2. 再做 `JSON -> 表单字段`
3. 再做 `答题提交`
4. 再做 `自动判分`
5. 最后做 `分享链接 + 公开访问`

这样最稳, 也最容易排错。

## 17. 最终建议

如果你现在就是想尽快做成:

- **推荐路线**: `Word 上传 -> 后端解析 -> 宜搭生成表单/页面 -> 发布分享链接 -> 用户答题 -> 自动判分`

如果你要做成长期课程考试系统:

- **更优路线**: `Word 上传 -> 后端解析 -> 题库存储 -> 固定答题页面动态渲染 -> 分享链接 -> 自动判分`
