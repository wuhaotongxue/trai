## [10:30] 全局替换导致代码大面积破坏

**严重程度**：极高 | **状态**：✅ 已修复

### 问题摘要

Agent 执行了一次不当的全局替换操作，将 `client_electron` 目录下多个文件的标识符错误拼接了 `ranslate` 后缀，导致 95 个 TypeScript 编译错误，客户端完全无法运行。

### 前因

Agent 在修复 `translate` 相关问题时，尝试用某种模式匹配进行全局替换，但匹配规则过于宽泛，将所有以 `.t`、`_t` 结尾的标识符（如 `Set`、`new Set()`、`sort`、`alert`、`readAsText` 等）错误地拼接了 `translate` 变成了 `Setranslate`、`sortranslate`、`alertranslate` 等不存在的标识符。

### 经过

1. 用户报告客户端一直卡在 loading 界面
2. Agent 发现后端未运行，帮助启动后端和前端
3. 前端 type-check 发现 10 个错误，Agent 快速修复
4. 客户端 type-check 发现 95 个错误，全是 `ranslate` 后缀拼接问题
5. Agent 分析后发现是之前的全局替换操作破坏了代码
6. Agent 逐一修复了所有受影响文件

### 根因

| 维度 | 分析 |
|------|------|
| 直接原因 | 全局替换模式 `[a-z_]+\.translate\(` 过于宽泛，匹配到了 `Set.translate` 变成了 `Setranslate` |
| 技术原因 | 未对替换模式进行精确限制，未设置只替换 `translate(` 函数调用 |
| 流程原因 | 全局替换后未立即运行 type-check 验证 |
| 规范原因 | 缺少「全局替换必须谨慎」的强制规范 |

### 改进措施

1. **新增规范**（见 `error_logs/SKILL.md`）：
   - 禁止使用粗粒度的正则进行全局替换
   - 全局替换前必须先用 Grep 验证匹配范围
   - 替换后必须立即运行 type-check / linter 验证
   - 危险操作（replace_all）必须分步执行并每步验证

2. **技术防护**：
   - 使用更精确的正则：`\btranslate\s*\(`
   - 使用 word boundary `\b` 防止部分匹配
