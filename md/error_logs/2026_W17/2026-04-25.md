## [02:54] ❌ 想当然：Git 编码问题误判，导致 README Changelog 丢失

**严重程度**：高 | **状态**：✅ 已修复

### 问题摘要

Git 仓库中 README.md 中文显示乱码的问题，实际上是 **PowerShell 终端编码问题**（GBK），而非 Git 仓库本身的问题。误判导致多次"修复编码"提交反而把原本正常的 Changelog 内容丢失了。

### 前因

用户问"之前的 README 更新记录去哪了"，我检查 Git 历史发现 README.md 提交记录中有大量编码乱码（`�?`）。

### 经过

1. **误判编码**：我看到 Git Bash 中显示中文为 `�?`，误以为是文件编码损坏
2. **多次"修复"**：创建 `README_clean.md` 和 `wuhao_readme.md` 替换原文件
3. **删除文件**：用户要求删除这些临时文件，我执行了 `del` 命令
4. **用户追问**：用户再次询问更新记录去哪了
5. **深入调查**：通过 WebFetch 从 GitHub 原始 URL 获取内容，发现：
   - GitHub 网页显示正常（UTF-8 渲染）
   - PowerShell 终端显示乱码是因为 Windows 默认 GBK 编码
   - 原始提交的 README 包含完整 Changelog
6. **真相大白**：根本问题不是仓库编码，而是终端编码差异
7. **恢复数据**：从 GitHub f761f27f 提交恢复了完整的 Changelog

### 根因

| 维度 | 分析 |
|------|------|
| **直接原因** | 未区分"终端显示乱码"和"文件编码损坏" |
| **想当然** | 看到乱码就认为是文件问题，未验证 GitHub 网页显示 |
| **技术盲区** | 对 Windows PowerShell/CMD 默认编码（GBK/CP936）与 Git UTF-8 的冲突不熟悉 |
| **过度修复** | 没有先确认问题就尝试修复，反而制造了新问题 |

### 改进措施

1. **验证优于修复** - 遇到编码问题，先用 `WebFetch` 查看 GitHub 原始内容，确认是否真的是文件损坏
2. **终端编码意识** - Windows 终端默认 GBK，与 Git 的 UTF-8 不同，显示乱码不代表文件损坏
3. **勿过度干预** - 不确定问题时先调查清楚，再决定是否修复
